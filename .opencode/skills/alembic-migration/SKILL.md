# Alembic Migration Skill

## Overview
The ModestWear backend uses SQLAlchemy 2.0 async with asyncpg. `alembic/versions/` is **empty** as of the last commit — schema has never been migrated. Every model change must be paired with a migration. This skill is the source of truth for that workflow.

## When to use this skill
- Adding or modifying any model in `backend/app/models/`
- Before running `alembic upgrade head` for the first time
- Investigating drift between `Base.metadata` and the live database
- Editing `backend/alembic/env.py`
- Recovering from a failed or partial migration

## Critical env.py configuration

The current `alembic/env.py` must support **async** with `asyncpg`. Required pieces:

```python
from logging.config import fileConfig
import asyncio
from sqlalchemy.engine import Connection
from sqlalchemy.ext.asyncio import async_engine_from_config
from alembic import context
from app.config import settings
from app.models import Base  # imports all 12 models via __init__.py

config = context.config
config.set_main_option("sqlalchemy.url", settings.DATABASE_URL)

if config.config_file_name is not None:
    fileConfig(config.config_file_name)

target_metadata = Base.metadata

def do_run_migrations(connection: Connection) -> None:
    context.configure(connection=connection, target_metadata=target_metadata)
    with context.begin_transaction():
        context.run_migrations()

async def run_async_migrations() -> None:
    connectable = async_engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.",
    )
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()

def run_migrations_online() -> None:
    asyncio.run(run_async_migrations())

run_migrations_online()
```

## Authoring a new migration

```bash
cd backend
source venv/bin/activate
alembic revision --autogenerate -m "add product tags index"
```

**Always review the autogenerated file** at `alembic/versions/<rev>_<message>.py` before committing. Common autogenerate gotchas:

- **Enums** (e.g. `Order.status`, `Variant.size`) — SQLAlchemy cannot detect enum value changes. Hand-edit.
- **Server defaults** like `server_default=func.now()` render fine but check `server_default="0"` on `used_count` matches the model.
- **Indexes on `Product.slug`, `User.email`, `Variant.sku`** — must be `unique=True` in the model AND `op.create_index(..., unique=True)` in the migration.
- **JSONB columns** (`Product.tags`, `Order.shipping_address`, `Variant.images`) — use `sa.dialects.postgresql.JSONB`, not generic `JSON`.
- **`Numeric` for prices** (variant.price) — never `Float`. Migrations must use `postgresql.NUMERIC(10, 2)`.

## Initial baseline migration (one-time)

Since `versions/` is empty and the database likely has no schema, generate the full baseline:

```bash
alembic revision --autogenerate -m "initial schema: 12 models"
```

This will produce one large migration covering: users, addresses, categories, products, variants, orders, order_items, coupons, cart_items, wishlist_items, audit_logs, chatbot_logs. After reviewing, run `alembic upgrade head` and commit.

## Model file order in __init__.py

The current order is correct (base → user → address → category → product → variant → order → order_item → coupon → cart_item → wishlist_item → audit_log → chatbot_log). **Do not reorder** — Alembic autogenerate walks imports alphabetically and reordering causes phantom diffs.

## Common commands

| Task | Command |
|---|---|
| Apply all pending | `alembic upgrade head` |
| Roll back one | `alembic downgrade -1` |
| Roll back to base | `alembic downgrade base` |
| Show current rev | `alembic current` |
| Show history | `alembic history --verbose` |
| Generate empty (hand-write) | `alembic revision -m "..."` |
| Generate from model diff | `alembic revision --autogenerate -m "..."` |

## Things to never do

- Never edit a migration after it has been applied to a shared environment. Add a new one.
- Never delete a migration file from `versions/`. Squash only at major version boundaries.
- Never run `alembic upgrade head` in production without a backup — confirm `alembic current` first.
- Never import a model inside a migration. Migrations must be self-contained.
- Never use `--sql` mode in development; it's only for offline SQL generation (e.g., DBA review).

## Relationship to `run.sh`

`cmd_backend_dev` and `cmd_dev_tmux` both call `alembic upgrade head` before starting uvicorn. If a migration fails, the backend will not start. Always run `alembic upgrade head` manually first to surface errors before the dev script does.
